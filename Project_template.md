## 1. Сравнение LLM-моделей (локальные Hugging Face vs облачные OpenAI, YandexGPT)

| Критерий | Hugging Face | OpenAI | YandexGPT |
| :--- | :--- | :--- | :--- |
| **Качество ответов (RAG)** | Потенциально идеальное, но требует инженерии. Можно выбрать модель под специфику документов. Риск галлюцинаций ниже при точной настройке. | Отличное для общего понимания. Высокий риск "галлюцинаций" на противоречивых данных. Плохо работает с узкими внутренними терминами. | Хорошее для русской документации. Адекватно работает с ФИО, должностями, аббревиатурами РФ. Меньше "отрывов" от контекста. |
| **Выявление пробелов** | Максимальная гибкость. Можно считать метрики неуверенности (perplexity) или использовать специальные модели оценки. Требуется собственная разработка аналитики. | Средняя. Анализ возможен только через логи: фиксация отказов и дизлайков. Нужно самостоятельно писать систему классификации вопросов. | Хорошая интеграция с Yandex Cloud. Возможность собирать статистику неотвеченных вопросов и строить дашборды в DataLens "из коробки". |
| **Скорость работы** | Зависит от инфраструктуры. Максимальная при локальном развертывании (On-premise). При аренде GPU — стабильно высокая, но дороже. | Быстро. Запросы идут через интернет, но задержки минимальны (100–300 мс). Зависит от географической удаленности дата-центров. | Быстро + локализация РФ. Низкий пинг для российских компаний. Отличная отзывчивость интерфейса. |
| **Стоимость владения** | High risk / High reward. Бесплатно при своих GPU. Дорого при аренде GPU 24/7. Требуется зарплата инженера для поддержки и дообучения. | OpEx (Pay per token). Плата только за использование. Выгодно при неравномерной нагрузке. Нет затрат на администрирование серверов. | OpEx (Pay per token). Цены обычно ниже OpenAI. Есть стартовые гранты. Простое прогнозирование бюджета через облако. |
| **Удобство развёртывания** | Сложно. Требуется выбрать эмбеддинги, LLM, написать оркестратор (LangChain или свой код), развернуть векторную БД, обеспечить отказоустойчивость. | Умеренно. OpenAI предоставляет эмбеддинги и LLM. Но всё равно нужно склеивать с векторной БД и писать логику. Нет готового корпоративного GUI. | Проще (в экосистеме РФ). Единая консоль Yandex Cloud. Есть готовые эмбеддинги и интеграция с сервисами поиска и аналитики. |
| **Безопасность данных** | Самый безопасный вариант при On-premise. Данные не покидают периметр компании. Требуется мощный IT-отдел. | Данные передаются на сервера в США/ЕС. Риски с точки зрения 152-ФЗ и коммерческой тайны. API может быть заблокировано. | Данные остаются в РФ. Яндекс имеет опыт работы с персональными данными. Проще получить сертификаты соответствия. |

## 2. Сраврнение моделей эмбеддингов (локальные Sentence-Transformers vs облачные OpenAI Embeddings):

| Критерий | Локальные (Sentence-Transformers / OSS) | Облачные (OpenAI Embeddings) |
| :--- | :--- | :--- |
| **Скорость создания индекса** | Зависит от GPU. На мощных GPU быстрее облака, требует настройки. На CPU — медленно. | Линейная, предсказуемая. Ограничена rate limits API. Не требует управления инфраструктурой. |
| **Качество поиска** | Высокое, особенно после fine-tune. BGE-large, NV-Embed сопоставимы с коммерческими. | Очень высокое. text-embedding-3-large — лидер бенчмарков. text-embedding-3-small — баланс скорости и качества. |
| **Стоимость владения** | Низкая OpEx, высокий CapEx. Модели бесплатны. Затраты на GPU (аренда $0.5–3/час) и инженеров. Окупается при больших масштабах. | OpEx (pay-per-token). small: $0.02/1M токенов, large: $0.13/1M токенов. Нет затрат на инфраструктуру. Выгодно для MVP и средних объёмов. |

OpenAI text-embedding-3-large на старте проекта может дать болеею точную "картину реального покрытия" базы знаний, без искажений, вызванных недостаточной точностью поиска.

## 3. Сравнение векторных баз ChromaDB и FAISS

| Критерий | ChromaDB | FAISS |
| :--- | :--- | :--- |
| **Скорость поиска и индексации** | Достаточная для малых и средних наборов данных (до 100 тыс. документов). На CPU показывает хорошую производительность, но уступает FAISS в скорости индексации и поиска. | Абсолютный лидер по скорости. Поддержка GPU даёт значительное преимущество. Лучший выбор для миллионных коллекций и требований к субмиллисекундной задержке. |
| **Сложность внедрения и поддержки** | Простая установка и настройка. Готова к использованию сразу, есть встроенное управление персистентностью, метаданными и фильтрацией. Идеально для быстрого старта. | Высокая сложность. Это библиотека, а не полноценная БД. Требует ручного управления индексами, памятью и сериализацией. Необходима экспертиза MLOps для продакшена. |
| **Удобство в работе** | Высокое. Интуитивный Python API, есть методы CRUD, поддержка фильтрации по метаданным из коробки. Хорошо интегрируется с LangChain и LlamaIndex. | Низкое / среднее. Нет встроенного управления метаданными. Требует ручного преобразования данных и внешних решений для хранения метаинформации. Сфокусирована только на векторном поиске. |
| **Стоимость владения** | Бесплатная (open-source). Работает на CPU, что снижает затраты на инфраструктуру. Оптимальна для проектов до сотен тысяч векторов. | Бесплатная (open-source). Однако для серьёзных нагрузок требуются GPU-инстансы, что увеличивает стоимость. Память — ограничивающий фактор, так как векторы хранятся в RAM. |

## 4. Выбор рекомендуемой конфигурации сервера (CPU, RAM, GPU), чтобы развернуть RAG-бота.

На старте проекта можно выбрать конфигурацию для сервера, не использующую GPU (однако желательно, что бы сервер позволил поставить GPU позже):

- CPU 8+ ядер
- RAM: 32+ GB
- Диск SSD 1 TB
- Векторная БД: ChromaDB или FAISS
- Эмбеддинги: локальные (sentence-transformers) или облачные (YandexGPT/OpenAI)
- LLM: облачная (YandexGPT/OpenAI) для быстрого старта

После запуска MVP необходимо мониторить нагрузку, ключевым фактором является время поиска.
